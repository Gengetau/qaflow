Date: 2026-07-01

## 1. 项目概述

QAFlow 是一个前后端完整的测试管理 Web 应用，用于管理测试项目、测试套件、测试用例、测试执行、缺陷追踪、证据附件和测试报告。

项目定位是一个面向中小团队或个人 QA/开发者的轻量级测试管理平台，类似精简版 TestRail + Jira Defect Board，但重点是展示完整的企业级 Web 应用工程能力。

QAFlow 不是一个简单的 Todo 应用，也不是只做页面展示的前端模板。它应该具备真实业务流程、权限控制、数据建模、状态流转、文件上传、统计 dashboard、报表导出、Docker 本地部署和自动化测试。

## 2. 项目目标

QAFlow 的目标是展示一个可运行、可维护、可扩展的全栈工程项目。

核心展示点包括：

1. Vue 3 + TypeScript 的现代前端工程能力。
2. Java 21 + Spring Boot 的企业级后端开发能力。
3. PostgreSQL + Flyway 的关系型数据建模和迁移能力。
4. Spring Security + JWT 的认证与权限控制能力。
5. 测试用例、测试运行、缺陷、附件和报告之间的业务闭环。
6. REST API + OpenAPI 文档 + 前端类型约束。
7. Docker Compose 一键启动。
8. 后端单元/集成测试、前端测试、E2E 测试和 CI 质量门禁。
9. 清晰的 README、架构文档和演示数据。

## 3. 为什么选择这个项目

这个项目适合作为作品集，因为它同时具备前端展示效果和后端业务复杂度。

普通 CRUD 项目很难体现工程能力，但 QAFlow 有天然的状态机、权限、报表、附件、审计日志和统计数据。

它也贴合实际工作场景：测试规格书、测试执行记录、缺陷管理、截图证据和报告导出都是企业项目中常见的需求。

项目完成后，可以在作品集中这样描述：

> Built a full-stack QA management platform with Vue 3 and Java 21 Spring Boot, supporting test case management, test run execution, defect tracking, evidence attachments, dashboards, report export, RBAC, PostgreSQL persistence, Docker Compose, and automated tests.

### 4.2 版本策略

后端使用 Java 21 LTS。Oracle 的 Java SE 支持路线中，Java SE 21 是 LTS 版本，Premier Support 到 2028 年，Extended Support 到 2031 年，因此它比追最新非 LTS 更适合作品集里的企业级后端项目。

Spring Boot 选择 3.5.x 稳定线，而不是一开始追 4.x。Spring Boot 官方文档当前列出了 4.1.0、4.0.7、3.5.15、3.4.13 等稳定版本；本项目选择 3.5.x 是为了保持 Java 21 兼容、生态稳定和 Codex 实现友好。

Maven 使用 Maven Wrapper 固定 3.9.x。Maven 官方历史页面说明 Maven 3.8.9 及更早版本已经 EOL，新插件发布会要求 Maven 3.9.0 或更高版本，因此本项目不使用旧 Maven 版本。

前端使用 Node.js 24 LTS。Node 官方发布说明建议生产应用只使用 Active LTS 或 Maintenance LTS，而 Node 24 当前处于 LTS 状态，所以前端工程锁定 Node 24。

```
Java: 21
Spring Boot: 3.5.x
Maven: 3.9.x via Maven Wrapper
Node.js: 24 LTS
pnpm: 10.x
PostgreSQL: 16 or 17
```

## 6. 核心业务流程

### 6.1 测试管理主流程

```
User registers / logs in
 -> creates workspace
 -> creates project
 -> creates test suite
 -> writes test cases and steps
 -> creates test run
 -> executes test cases
 -> marks results as Passed / Failed / Blocked / Skipped
 -> creates defects from failed test cases
 -> attaches screenshots or evidence
 -> tracks defect status
 -> exports test report
 -> dashboard updates automatically
```

### 6.2 缺陷处理流程

```
Failed test run item
 -> create defect
 -> assign severity and priority
 -> attach evidence
 -> set status Open
 -> move to In Progress
 -> move to Resolved
 -> verify fix
 -> close defect
```

### 6.3 报告流程

```
Project selected
 -> date range selected
 -> collect test run results
 -> aggregate pass rate and defect status
 -> generate report
 -> export as PDF or HTML
```

## 7. 用户角色与权限

### 7.1 角色定义

| Role | 权限 |
| ------ | ---------------------------------------------- |
| OWNER | 管理 workspace、项目、成员、删除数据、生成报告 |
| TESTER | 创建测试用例、执行测试、创建缺陷、上传附件 |
| VIEWER | 查看项目、测试结果、缺陷和报告 |

### 7.2 权限规则

OWNER 可以管理所有项目资源。

TESTER 可以参与项目执行，但不能删除项目或管理成员。

VIEWER 只能读取数据，不能写入测试结果、缺陷或附件。

所有 API 都必须校验当前用户是否属于目标 workspace，并根据角色判断操作权限。

## 17. 安全设计

### 17.1 Authentication

使用 JWT access token 和 refresh token。

Access token 生命周期短，例如 15 分钟。

Refresh token 生命周期较长，例如 7 天，并存储 hash 到数据库。

### 17.2 Authorization

所有项目资源都通过 workspace membership 校验。

所有写操作都必须校验用户角色。

### 17.3 Password

密码使用 BCrypt 哈希。

注册和登录接口需要基础 rate limit，v1 可先在文档标注，后续实现。

### 17.4 File Security

附件下载必须校验权限。

用户不能通过 storage path 直接访问文件。

文件名需要清洗，防止 path traversal。

## 18. 数据库迁移策略

使用 Flyway 管理数据库迁移。

JPA 配置使用 validate，不使用 update 自动改表。

```
spring:
 jpa:
 hibernate:
 ddl-auto: validate
 flyway:
 enabled: true
```

迁移文件命名：

```
V1__init_schema.sql
V2__test_management.sql
V3__test_runs.sql
V4__defects_and_attachments.sql
V5__reports_and_activity.sql
```

数据库约束必须覆盖关键业务规则：

```
unique workspace slug
unique project key within workspace
unique test case key within project
foreign key constraints
not null constraints
status check constraints where practical
indexes for list and dashboard queries
```

## 26. 成功标准

这个项目成功的标准是：

1. 评审可以 5 分钟内理解它是一个完整 QA 管理平台。

10. 项目可以写进简历和作品集。

## 27. 推荐最终定位

QAFlow 应该被包装成一个企业内部 QA 管理工具，而不是普通测试用例 CRUD。

作品集描述应突出：

```
Full-stack SaaS-style QA management platform
Java 21 + Spring Boot backend
Vue 3 + TypeScript frontend
PostgreSQL + Flyway
JWT + RBAC
Test execution workflow
Defect tracking
Evidence attachments
Dashboard and reports
Docker + CI + automated tests
```

最终目标是让评审看到主人不仅会写页面和接口，还能设计完整业务流程、权限模型、数据模型、测试策略和工程交付路径。
